# Front-End Page Structure

A production page should be organized around predictable regions: header, main content, supporting navigation, related resources, and footer. The `main` element should contain the unique page content, while repeated navigation and footer blocks stay outside it. This makes the document easier for screen readers, search crawlers, and test automation to understand. For guide-style pages, the recommended structure is an intro block, a feature summary, a step list or comparison area, an FAQ section, and a final related-link group.

```html
<header class="site-header">...</header>
<main id="content">
  <section class="page-intro">...</section>
  <section class="content-grid">...</section>
  <section class="faq-list">...</section>
</main>
<footer class="site-footer">...</footer>
```

JavaScript should enhance the page without making basic reading dependent on client-side execution. Accordions, tabs, and filters can attach behavior after the HTML renders, but default content should remain accessible. When routing between pages, preserve scroll restoration, update the document title, and move focus to the main heading so keyboard users receive a clear transition. Images should include useful `alt` text, and decorative icons should be hidden from assistive technology.

The page shell should also provide a consistent container width and spacing scale. Keep content groups visually separate with headings and whitespace instead of relying on heavy borders everywhere.
